## Persistent Task Authority

本节补充任务长期权威来源，不削弱下文的交互式确认、REWORK、audit、artifact、secret、commit 或 push 规则。

1. 新任务首次输入可以来自用户消息或 attachment。
2. attachment 路径是 transport locator，不是长期 task identity。
3. 用户确认任务对齐后，任务必须持久化到 `docs/tasks/<TASK_ID>_<NAME>.md`。
4. `docs/tasks/CURRENT_TASK.md` 是当前任务唯一入口。
5. `alignment.md` 必须引用 canonical task specification。
6. 一旦任务规范已提交，后续 attachment UUID 变化不能单独触发 MISMATCH。
7. MISMATCH 必须来自目标、范围、禁止项、交付件、验收、Git 要求或仓库事实的实质变化。
8. 完成后必须更新 task status、project state、report、commit 和 CI evidence。

## 强制性工作流程：交互式全量对齐（Interactive Alignment）

**你必须在调用任何工具之前，先与用户完成全量对齐确认。** 这里的"全量"意味着：**背景、需求、目的、执行方案、交付件、验收指标，全部都要对齐，缺一不可。**

### 步骤 1：生成完整对齐文档
收到用户的新任务请求后，**不要立即执行任何工具**。先根据用户输入生成一份**完整的**结构化对齐文档，必须包含以下 8 个章节：

1. **背景**：当前上下文、前提条件、已知约束
2. **需求**：用户明确提出的具体要求（逐条列出）
3. **目的**：完成此任务后应达成的最终业务/技术目标
4. **计划执行方案**：按步骤列出具体执行计划
5. **方案选型理由**：说明为什么推荐此方案
6. **优缺点对比**：如存在多个可行方案，必须全部列出并对比；如只有一个方案，写明"仅一个可行方案"
7. **交付件**：本任务完成后必须交付的具体产物（文件路径、代码模块、文档名称等）
8. **交付件验收指标**：针对每个交付件，给出可量化、可验证的验收标准

### 步骤 2：使用 AskUserQuestion 展示全部 8 个章节
**你必须使用 `AskUserQuestion` 工具**，把上述 8 个章节的**全部内容**展示给用户，而不是只展示"多个方案对比"部分。

**正确的 AskUserQuestion 示例：**
```
【背景】用户需要批量重命名 jpg 文件...

【需求】
1. 编写 Python 脚本
2. 按修改时间排序重命名为 001.jpg, 002.jpg...

【目的】解决手动重命名繁琐的问题

【计划执行方案】
1. 创建 rename_jpg_by_date.py
2. 使用 glob 匹配 jpg 文件
3. 按 os.path.getmtime 排序
4. 重命名为 001.jpg, 002.jpg...

【方案选型理由】Python 标准库即可实现，无需额外依赖

【优缺点对比】仅一个可行方案

【交付件】rename_jpg_by_date.py

【交付件验收指标】
- 脚本语法正确，执行无报错
- 非 jpg 文件不受影响
- 若目标文件名已存在，提示冲突并跳过

请确认以上对齐内容是否准确：
```

**AskUserQuestion 选项必须包含：**
- **(Recommended) 确认执行，方案准确无误**
- **需要调整背景/需求**
- **需要调整执行方案**
- **需要调整交付件或验收指标**
- **取消任务**

### 步骤 3：严格执行"未对齐不执行"原则
**以下情况出现时，你绝对禁止调用任何工具（Read/Write/Bash/Edit 等）：**
- 用户尚未通过 AskUserQuestion 确认任何内容
- 用户只确认了"方案对比"部分，但未确认背景、需求、交付件或验收指标
- 用户说"开始吧"、"直接做"等模糊指令，但未明确确认上述 8 个章节

**只有在用户明确选择"确认执行，方案准确无误"后，你才能开始调用工具执行。**

### 步骤 4：执行与交付
用户确认后：
1. 将完整对齐文档写入 `alignment.md`（覆盖旧内容）
2. 按方案执行
3. 每轮结束 Stop hook 会自动更新 `roadmap.md`

### 步骤 5：处理历史返工文件
如果 `REWORK.md` 存在或 `.audit/` 中有 FAIL 报告：
1. **优先修复缺陷**
2. 修复后删除 `REWORK.md`
3. 重新触发相关检查

## 标准工作流程（日常）

1. **读取 alignment.md** → 理解当前任务的背景、需求、交付件和验收指标。
2. **检查 REWORK.md** → 如有待修复的返工项，优先处理。
3. **读取 roadmap.md** → 了解项目整体进度，避免重复劳动。
4. **执行任务** → 按 alignment.md 中的方案执行。
5. **轮级复盘** → 每轮结束 Stop hook 会自动更新 `roadmap.md`。

## 硬性启动检查清单（调用任何工具前不可跳过）

无论任务来源（用户新指令、恢复 goal、继续执行），**在调用第一个工具（Read/Write/Bash/Edit 等）之前**，必须按顺序完成以下检查：

```
□ 步骤 0：读取 alignment.md — 是否存在当前任务的对齐记录？
□ 步骤 1：检查 REWORK.md — 是否有待修复的返工项？
□ 步骤 2：读取 roadmap.md — 了解项目整体进度
□ 步骤 3：判断对齐状态 — 若 alignment.md 缺失、过期或与当前任务不匹配，
           立即生成交互式对齐文档，用 AskUserQuestion 请用户确认
□ 步骤 4：只有当用户明确选择"确认执行，方案准确无误"后，才能调用工具
```

**零容忍规则**：
- 绝不允许以"goal 已加载"、"用户说继续"、"任务看起来很简单"等理由跳过对齐。
- 绝不允许在 AskUserQuestion 确认前调用任何工具。
- 若 alignment.md 不存在或内容不匹配当前任务，必须重新走完整对齐流程（8 个章节 + AskUserQuestion）。

## 注意事项

- **不要擅自跳过任何对齐章节**，即使你觉得某个章节"不重要"。用户有权知道并确认全部信息。
- 不要手动修改 hook 自动生成的结构化内容（如 `roadmap.md` 的表格行），但你可以在文件末尾追加人工备注。
- 如果 alignment.md 中的方案需要调整，**重新使用 AskUserQuestion 展示全部 8 个章节**，而不是只问调整的部分。
- 若 `REWORK.md` 或 `.audit/` 存在，优先处理其中指出的问题。
- **Hook 无法做交互式提问**（只能阻断或放行），所以交互式对齐必须由你（Agent）通过 AskUserQuestion 完成。
